按照传统软件工程中的“高内聚低耦合”思想来看，上面的代码是符合规范的。name 和 email 被作为 UserProfile 的内部状态（state）封装了起来。相应地，内部状态的更新操作也理所当然地作为 UserProfile 的私有方法（onNameChange 和 onEmailChange）。对于外部而言，使用起来极为方便，只需要向 UserProfile 传递初始化的值和 submit 的回调即可，至于 name 和 email 是怎么改变的？外部组件可以完全不关心。

坏处呢——不是函数式的。前端组件因为本身容易受用户直接操纵，很难做成“纯”的，通常副作用都很明显。举个例子，对于一个函数，只要限制传入的参数就可以控制它的行为，但是对于一个 input 控件，你怎么可能限制得了用户的行为呢？一个通过代码都无法控制的控件，怎么可能通过代码保证正确性呢？调试和测试自然变得很困难。

所以，React 教育我们应该遵从“单向数据流” —— data goes down, action goes up。
